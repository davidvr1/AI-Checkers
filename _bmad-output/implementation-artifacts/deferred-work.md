# Deferred Work

## Deferred from: code review of spec-checkers-ai-opponent (2026-07-22)

- Crowning mid-jump doesn't end the turn (`src/game/rules.ts` / `applyMove` continuation logic). The shared rules engine (introduced in the base-game story, a documented judgment call at the time) lets a freshly-promoted man continue capturing as a king in the same turn, and the minimax AI now actively seeks out and exploits these lines. Pre-existing shared-engine behavior, not caused by any single feature. **Updated 2026-07-22 (Israeli-draughts renegotiation):** the earlier rationale referenced "Standard American/English draughts ends a move on crowning"; the shipped ruleset is now Israeli/international, under which a man that merely passes through the last row during a capture is typically *not* crowned (crowning happens only if the move ends there). The new man-vs-king forward-only capture restriction is also silently bypassed on the leg after a mid-chain promotion (the continuation runs the unrestricted king branch). Still needs a focused rules-correctness decision: confirm the intended promotion-ends-turn rule and, if changed, update `applyMove`'s continuation check plus the affected tests.

## Deferred from: online-multiplayer review (Blind Hunter + Edge Case Hunter, 2026-07-23)

Fixed in the feature: the malformed-move server crash (shape validation in `GameRoom.move` + try/catch + `uncaughtException` guard), interactive-board-after-drop (`myTurn` now requires an open socket), no keepalive (ping/pong heartbeat reaps ghost seats), no reconnection (client auto-reconnects with backoff), over-aggressive auto-reset (an in-progress game is now preserved when both seats empty), `maxPayload` cap, SPA 404 for missing assets, and the multi-jump rapid-click race (a pending-move lock). Deferred:

- **Spectator auto-promotion.** When a seated player leaves mid-game and a spectator is already connected, the freed seat is not auto-assigned to the spectator; they must reload to claim it (`src/net/gameRoom.ts` `leave`/`join`). Fine for the 2-player home case; revisit if spectators become common. Proper fix: on `leave`, promote the oldest connected spectator and send it a fresh `welcome`.
- **Color preservation across a simultaneous double-drop.** If BOTH players drop mid-game and reconnect in the opposite order, they may swap Red/Black (seats are assigned by join order, not identity). Single-drop reconnection preserves color correctly (the remaining player holds their seat). Proper fix needs per-client session tokens.
- **Mid-game board inherited by a new unrelated pair.** Since an in-progress game is preserved on empty (so transient double-drops resume), a genuinely abandoned mid-game persists; a fresh unrelated pair joining the same server inherits it and can only reset once it reaches game-over (Rematch is game-over-only). Rare on a home server used by the same two people.
- **No rate limiting / no auth.** A seated LAN client can spam `move`/`reset` (each triggers a full broadcast); the server trusts everyone on the WiFi. Acceptable for a private home network; not internet-safe.
- **Dev-only StrictMode double-connect.** Under `npm run dev` + React StrictMode the hook connects→disconnects→connects, which can transiently grab both seats. The *served* production build (`npm run serve`) does not double-invoke effects, so real play is unaffected; only local `vite dev` testing is misleading.

## Deferred from: code review of spec-checkers-game (Israeli-draughts ruleset, 2026-07-22)

- Immediate capture removal vs. international "phantom blocker" rule (`src/game/rules.ts` `applyMoveToBoard` + `gameReducer.ts` continuation scan). Captured pieces are removed from the board immediately, so a flying king's multi-jump continuation is computed on the already-thinned board; strict FMJD/international rules keep captured pieces in place as blockers until the whole sequence ends (you may not jump the same piece twice, nor re-cross a square you vacated mid-chain). Deferred: the source rules page (he.wikibooks Dama) does not specify phantom-blocking, and the shipped simplification ("land on any empty square beyond the first enemy piece") is a reasonable reading; revisit only if strict international capture-blocking is later desired.
- Threefold-repetition search pollution (`src/ai/minimax.ts` `chooseAiMove`/`search` + `gameReducer.ts` `applyMove`). Because minimax calls the shared `applyMove`, every hypothetical line inherits and extends the *real* game's `positionHistory`, so a searched endgame board that coincides with a key already seen twice in real play is scored as a draw (0) even though the searched line did not itself repeat three times. Endgame-only and subtle. Cheap potential fix if it ever matters: seed the search root with an empty `positionHistory` so repetition is detected only within the searched line.
- AI hot-path constant-factor growth (`src/game/board.ts` `positionKey`, `gameReducer.ts` `applyMove`, `rules.ts` `isInsufficientMaterial`). Every applied move — including every minimax node up to the 200k-node cap — now builds a ~65-char position key, spreads/filters the `positionHistory` array, and (in king-heavy endgames) full-scans the board for insufficient material, where the old engine did an integer increment. Bounded by the existing `SEARCH_DEADLINE_MS` wall-clock cutoff (the search returns best-so-far), so latency is capped, but effective depth under flying kings' larger branching factor is reduced and unmeasured. Revisit with a re-benchmark if AI strength feels low; a transposition table or an incremental Zobrist key would address it.

## Deferred from: code review of the move-log feature (2026-07-22)

- `GameState.history` grows via a plain array spread with no cap and no append-only enforcement (`gameReducer.ts`'s `withHistory`, `moveLog.ts`'s `index: state.history.length` numbering). Harmless at this game's real scale (a full game is at most a few dozen real moves, bounded further by the 40-turn draw rule), and nothing currently mutates history non-append-only. Would need real enforcement (or at least a test pinning the assumption) only if a future feature -- undo, save/load replay, move-list scrubbing -- starts mutating or truncating `history` directly.
